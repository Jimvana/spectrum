🔬 .spec file anatomy

bytes 0–3 Magic: SPEC — identifies the format, like a fingerprint
bytes 4–5 Dict version — which token dictionary was used to encode
bytes 6–7 Flags — e.g. bit 0 = RLE compression was applied
bytes 8–11 Original length — exact byte count of source, for verification
bytes 12–13 Language ID — Python / HTML / JS / CSS / Text
bytes 14–15 Checksum — sum of all original bytes mod 65536 (fidelity proof)
bytes 16+ Body — zlib-compressed stream of uint32 token IDs with RLE runs collapsed

⚙️ How encoding works, step by step

1
Tokenise — source is split into meaningful tokens using language-aware tokenisers (Python keywords, HTML tags, JS operators, CSS selectors, English words)
2
Map to IDs — each token maps to a uint32 ID from the shared Spectrum dictionary (234K+ entries). Unknown chars fall back to ASCII or Unicode escape IDs.
3
RLE pass — runs of 3+ identical token IDs are collapsed: ID, RLE_MARKER, count. Saves significantly on indented code and repeated patterns.
4
zlib level 9 — the ID stream is compressed. Because IDs are semantically grouped (similar tokens have nearby IDs), DEFLATE compresses them harder than raw bytes.
5
16-byte header is prepended. Result is a self-contained, versioned, verifiable .spec file.

❌ Traditional RAG — what it actually stores

📄
Raw text chunks (full size) e.g. 1 MB of source code stored as-is, uncompressed
🔢
Float embeddings (per chunk) 1536 float32 values × 4 bytes = 6 KB per chunk. 1000 chunks = 6 MB just in floats — on top of the text
🌐
External embedding model required Every new document, every query — paid API call or local model run
📊
Cosine similarity on floats Meaningful but opaque — you can't inspect or reason about why two chunks matched

✅ Spectrum RAG — what it stores instead

📦
.spec files only — text baked in, compressed That same 1 MB source code → 260–300 KB .spec. No separate float arrays.
🧩
Token IDs are the "embedding" The token frequency distribution across the .spec body IS a semantic fingerprint — no external model needed to generate it
🔌
No external dependencies at query time Spectrum encoder/decoder is self-contained — works offline, no API cost
🔍
Interpretable similarity You can see which tokens drove a match. "These chunks matched because they both heavily use class, def, import tokens."

💡 The core insight

Traditional RAG stores your data twice — once as readable text, once as a huge opaque blob of floats. The floats are generated by an external model that has no knowledge of your specific data's structure. They can't be inspected, don't compress well, and cost money to generate.

Spectrum's token IDs are different. They're semantically grounded — they represent actual meaningful units of your source language (Python keywords, HTML tags, JS operators, English words). Two files that share many of the same token IDs are genuinely similar in structure and content. You don't need a neural network to tell you that — the token overlap is the signal.

The open question is: is token-ID similarity good enough to replace vector embeddings for retrieval? For code and structured text, it very well might be — and it would be faster, cheaper, smaller, and fully explainable. That's Phase 2.

🔬 Three ways token IDs could power retrieval

Token Frequency (TF-IDF style)

Count how often each token ID appears in a .spec chunk. Build a sparse vector of token frequencies. Use cosine similarity on those sparse vectors — no neural net, no floats. Mathematically equivalent to BM25 but using Spectrum tokens instead of words.

Token Sequence (n-gram)

Look at sequences of token IDs, not just frequencies. Two code files that share the same patterns of token IDs (e.g. def → name → ( → args → )) are structurally similar. This catches things frequency alone misses.

Token Fingerprint (MinHash)

Use the set of unique token IDs as a MinHash signature — a fixed-size fingerprint of the content. Ultra-fast approximate similarity search with no floating point at all. Scales to millions of .spec files.
